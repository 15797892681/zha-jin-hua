# 单机 AI 高手模式设计

- 日期：2026-07-14
- 状态：方案已确认，书面设计已自检，待用户复核
- 范围：单机模式的三名 AI 玩家

## 1. 背景与根因

当前线上单机 AI 有两条决策路径：DeepSeek 成功时采用模型返回的动作，调用失败时采用浏览器内的 `chooseAiAction`。现有 DeepSeek 提示词只约束人设、返回格式和动作合法性，没有牌力评估、底池赔率、连续加注识别或反诈唬约束。因此，一个动作虽然合法，也可能在策略上明显过弱。

上一轮新增的平衡测试只模拟了本地 fallback，而且模拟时没有传入真实单机模式维护的公开记忆。它能证明规则 AI 有所改善，但不能代表线上占主要比例的 DeepSeek 决策链。生产日志抽样也确认，大多数回合使用的是 DeepSeek，而不是 fallback。

本次改造选择“共享战术引擎 + DeepSeek 人设”方案：战术判断由可测试、可复现的共享代码负责，DeepSeek 只在战术允许的低风险候选动作中体现人物差异并生成台词。

## 2. 目标与非目标

### 2.1 目标

- 连续机械加注不再成为稳定获胜策略。
- DeepSeek 成功和本地 fallback 两条路径遵守同一套战术下限。
- AI 看牌后基于真实牌力胜率、底池赔率、剩余对手数和近期进攻性决策。
- AI 能识别主要加注者，并在合适牌力下优先跟注、反加或向其发起比牌。
- 保留谨慎、激进、混沌三种人设，但人设只能在合理候选动作之间产生差异。
- 弱牌仍能弃牌，避免把高手模式做成无脑跟注或三名 AI 联手针对玩家。
- 所有决策继续经过现有合法动作校验和 `applyAction`，不修改核心游戏规则。

### 2.2 非目标

- 不引入作弊信息；未看牌的 AI 不能使用自己的底牌，任何 AI 都不能看到对手底牌或牌堆顺序。
- 不实现完整博弈树、强化学习或在线自学习。
- 不改变在线真人模式、下注档位、结算规则或筹码规则。
- 不保证玩家必输；高手模式仍应允许合理策略获胜。

## 3. 验收标准

- 在至少 10,000 个固定种子的四人局中，玩家始终不看牌并机械选择下一档加注，玩家胜率不高于 26%，且平均结束筹码低于初始筹码 1000。
- 四人公平随机发牌下，即使所有 AI 都坚持比牌，玩家拿到全场最强牌的理论概率仍约为 25%；因此不使用低于 25% 的胜率目标，避免倒逼 AI 共享隐藏牌或获得发牌优势。
- 平衡模拟必须使用与 `useSoloGame` 相同的公开记忆写入方式，并覆盖三种 AI 人设。
- 模拟中的每个 AI 动作都必须来自共享战术引擎允许的候选集，且所有牌局在动作上限内结束。
- 面对近期连续加注者：
  - 谨慎型未看牌 AI 优先看牌后判断牌力；
  - 激进型和随机型 AI 保留盲注身份，通过盲加或盲跟抵消看牌后的双倍下注成本；
  - 中强牌不得因单次随机数或人设直接弃牌；
  - 强牌在具备合法条件时优先反加或向主要加注者比牌。
- 明显弱牌且跟注或比牌期望不合理时仍允许弃牌。
- 服务端集成测试必须证明 DeepSeek 看到的是战术收窄后的动作集合，而不是原始全部合法动作。

## 4. 总体架构

新增一个纯函数战术层，放在共享目录并同时被服务端 DeepSeek 路径与浏览器 fallback 使用：

```text
GameState + public memory
  -> buildAiDecisionRequest
  -> buildTacticalPolicy
       -> 牌力胜率
       -> 底池赔率
       -> 对手数量
       -> 主要加注者与压力等级
       -> 人设阈值修正
  -> rankedActions + safeActions
       -> DeepSeek：只能从 safeActions 选择并生成台词
       -> fallback：执行 rankedActions[0] 并使用本地台词
  -> 原始合法动作复核
  -> applyAction
```

战术引擎不修改状态，也不执行动作。它只根据脱敏的 `AiDecisionRequest` 返回评估结果和候选动作。这样服务端不需要获得其他玩家底牌或完整牌堆，浏览器也不会维护第二套策略。

## 5. 战术评估

### 5.1 牌力胜率

AI 未看牌时，`self.cards` 为 `null`，战术层不得估算或使用其真实底牌，只能根据公开压力、筹码和人设决定看牌、跟注或有限度施压。

AI 看牌后，使用标准 52 张牌的全部三张牌组合建立一次性牌力分布表，并用现有 `evaluateHand` 的牌型和比点规则计算当前手牌在该分布中的百分位，作为单个随机对手的胜率估计。分布表在模块内缓存，单次决策只做牌面规范化和查表，保证 10,000 局回归测试与浏览器交互的性能稳定。

多名活跃对手时，以单挑胜率和活跃对手数估算多方胜率。该估算只用于动作分层，不替代游戏结算，也不假设知道对手真实牌面。

### 5.2 价格与风险

战术层使用以下公开信息：

- `callCost / (pot + callCost)` 作为跟注所需的最低收益比例；
- `compareCost / (pot + compareCost)` 作为立即比牌的成本参考；
- 当前筹码与当轮投入，避免为了小底池投入不成比例的筹码；
- 活跃对手数，避免把单挑强度直接套到多人局；
- 动作次数，长局仍保留强制推进和结束机制。

### 5.3 连续加注识别

公开记忆中的 `raise`、`call`、`look`、`fold` 与桌面上的 `roundContribution` 共同形成对手压力分。近期加注权重最高，高投入但没有近期加注只提供较小权重。战术层输出压力等级以及主要加注者 `aggressorId`。

跨小局保留的 8 条公开记忆用于识别玩家短期风格；当前小局的 `roundContribution` 和 `actionCount` 用于判断本局是否正在连续施压。这样既能学习近期打法，也不会仅凭上一小局的一次加注就错误地强制比牌。

### 5.4 人设边界

- 谨慎型：提高主动加注阈值，边缘牌更偏向跟注，但中强牌面对连续施压时不能直接弃牌。
- 激进型：更早反加，强牌优先施压或比牌。
- 混沌型：在分值接近的安全候选动作之间使用随机数选择，可以有限诈唬，但不能跨越战术层的弃牌或跟注底线。

人设只调整小范围阈值和安全候选动作的排序，不得绕过牌力、赔率与合法性判断。

## 6. 战术策略输出

新增 `TacticalPolicy`，至少包含：

```ts
interface TacticalPolicy {
  preferredAction: AiActionIntent;
  safeActions: AiActionIntent[];
  aggressorId: string | null;
  pressure: 'low' | 'medium' | 'high';
  equity: number | null;
}
```

- `preferredAction` 是 fallback 使用的首选动作。
- `safeActions` 是 DeepSeek 可以选择的完整集合，必须是原始合法动作的真子集或相同集合。
- 所有 `safeActions` 必须与首选动作处在相同战术等级，不能为了人设保留明显负期望动作。
- 当主要加注者仍是合法比牌目标时，面向反诈唬的 `compare` 优先选择该玩家。
- 如果战术计算因异常无法生成候选集，使用当前已有的保守合法回退：能跟则跟，否则弃牌；不得抛出导致牌局停滞的错误。

## 7. DeepSeek 与 fallback 数据流

### 7.1 DeepSeek 成功路径

1. 服务端校验原始 `AiDecisionRequest`。
2. 服务端调用共享战术引擎生成 `TacticalPolicy`。
3. 将请求中的 `legalActions` 收窄为 `safeActions` 能表达的集合，并把压力等级、主要加注者和不泄露牌面的策略说明加入系统提示词。
4. DeepSeek 只能从收窄后的动作集合中选择动作并生成对应人设台词。
5. 模型结果先对收窄集合校验，再对原始合法动作集合校验。
6. 客户端仍使用最新状态和 `applyAction` 做最终复核。

### 7.2 fallback 路径

客户端已经持有构建脱敏请求所需的信息。远程调用失败、超时、模型越过安全集合或响应过期时，客户端再次调用同一个共享战术引擎并执行 `preferredAction`。因此 provider 故障不会降低 AI 的基本战术能力。

模型越过 `safeActions` 被视为无效远程决策，沿用现有错误处理和熔断机制；不会静默执行策略上不允许的合法动作。

## 9. 测试设计

测试遵循红—绿—重构顺序：

1. 牌力胜率测试：同样压力下，豹子、顺金、对子、强散牌、弱散牌的百分位顺序正确，估值处于 `[0, 1]`。
2. 压力画像测试：近期多次加注且当轮投入最高的玩家被识别为主要加注者；只有旧局加注时不产生当前局高压误判。
3. 未看牌反制测试：高压下谨慎型优先看牌，激进型优先盲加，随机型在盲加和盲跟之间变化，封顶后盲跟。
4. 中强牌底线测试：面对连续加注，中强牌的 `safeActions` 不包含弃牌。
5. 强牌反击测试：合法时首选反加或向主要加注者比牌。
6. 弱牌纪律测试：赔率不合适时弱牌仍允许弃牌。
7. 人设边界测试：三种人设可以排序不同，但都不能返回安全集合外动作。
8. 服务端主链路测试：mock gateway 只能收到收窄后的动作集合；返回集合外动作会触发现有 fallback，而不会执行。
9. 平衡回归测试：至少 10,000 个固定种子、真实记忆、完整四人局，机械盲加玩家胜率不高于 26%，平均结束筹码低于 1000。
10. 全量回归：单元测试、类型检查、生产构建和 Playwright 测试全部通过。

## 10. 性能、容错与可观测性

- 三张牌牌力分布在进程内和页面内惰性构建并缓存；缓存只包含规则牌型分值，不包含用户信息或对局状态。
- 战术计算必须在调用 DeepSeek 前完成，但不得增加网络重试或新的外部依赖。
- 日志继续只记录请求 ID、耗时、模型、token 和状态；不记录牌面、胜率、完整战术上下文或用户打法详情。
- 战术策略生成异常时记录通用错误状态并使用保守合法回退，牌局不得停滞。
- 若性能测试表明首次构建分布影响交互，再把分布改为构建期静态表；在有证据前不提前增加生成脚本或新依赖。
